# v0.5.8 Gap Prioritization and Feature Scope Plan

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

## What this document is

This plan converts the gaps identified in the [Product Capability Inventory](product-capability-inventory.md) into a prioritized, scoped roadmap for the `v0.5.8` development cycle. It exists so reviewers and maintainers can see:

- What must be fixed before v0.5.8 is considered ready
- What should be fixed if time permits
- What is optional polish
- What is intentionally deferred
- What must never be built

This is **not** a commitment to implement everything listed. It is a planning and prioritization document.

## How gaps were derived

Gaps were derived from the product capability inventory by examining every capability with status:

- `partial` — works but with known limitations
- `experimental` — early stage, template output
- `disabled_by_default` — exists but off unless explicitly enabled
- `docs_only` — documented but no runtime enforcement
- `missing` — not implemented

Additional gaps were identified by reviewing public docs for overclaim risks, checker coverage, and reviewer onboarding friction.

## Priority definitions

| Priority | Meaning |
|---|---|
| `must_fix` | Blocks v0.5.8 readiness. Must be resolved before release. |
| `should_fix` | Improves release quality but does not block. |
| `could_fix` | Optional polish. Nice to have if time permits. |
| `defer` | Valid idea, but not for v0.5.8. May be revisited later. |
| `do_not_build` | Unsafe, out of scope, or contradicts safety posture. Never build. |

## Explicit non-goals for v0.5.8

The following are **not** goals for v0.5.8:

- Enabling live trading by default
- Expanding broker execution beyond Alpaca
- Unlocking provider execution runtime
- Adding autonomous trading loops
- Claiming profitability or trading correctness
- Supporting multi-user or cloud deployment
- Bypassing any safety gate, approval queue, or kill switch
- Sharing or facilitating credential exchange

## Safety-sensitive and deferred capabilities

Capabilities marked `safety_sensitive` or `protected_boundary` are deferred unless they are docs-only or checker-only changes:

| Capability | Status | v0.5.8 decision |
|---|---|---|
| Provider execution runtime unlock | deferred | Requires full safety review post-v0.5.8 |
| Live broker execution (non-Alpaca) | deferred | Requires adapter completion and safety review |
| Binance/CCXT/IBKR brokers | deferred | Partial/placeholder; not a blocker |
| Provider execution boundary docs | must_fix | Clarify docs only; no runtime change |

## Acceptance criteria for v0.5.8

v0.5.8 is considered ready when:

2. `./scripts/release_check.sh --quick` passes.
3. `scripts/smoke_reviewer_golden_path.py --json --skip-release-check` passes.
4. `scripts/build_release_evidence_bundle.py --skip-slow` passes.
5. `scripts/check_forbidden_claims.py` passes.
6. `scripts/check_public_launch_readiness.py` passes.
7. `scripts/check_stable_release_decision.py` passes.
8. `scripts/check_product_capability_inventory.py` passes.
9. `scripts/check_v058_gap_prioritization.py` passes.
10. No protected boundary changes are staged.
11. `main` remains at `0.5.8rc3`.
12. `v0.5.7` tag remains untouched.

## How reviewer feedback changes prioritization

Reviewer feedback can re-prioritize gaps through the [Feedback Triage Taxonomy](feedback-triage-taxonomy.md):

- **Blocker feedback** (breaks install, weakens safety, introduces forbidden claims) becomes a `must_fix`.
- **Non-blocker feedback** (docs clarity, UX polish) becomes a `should_fix` or `could_fix`.
- **Out-of-scope feedback** (live trading requests, profit evaluation) is rejected and documented.

When a blocker issue is opened:

1. Pause v0.5.8 scope work on non-blocker items.
2. Triage the blocker using the safety-sensitive path.
3. If accepted, add it to the must-fix list and update this document.
4. Re-run the release check.
5. Resume scope work only after the blocker is resolved.

## Safety posture summary

- **Live trading** is disabled by default.
- **Provider execution** remains locked (governed by artifact policy and risk manager, not a runtime API block).
- **Broker execution** remains blocked unless explicit opt-in gates pass.
- **No credentials** are required for default verification.
- **Not financial advice.** Does not imply profitability or trading correctness.
- **Not production ready** for unattended or real-money trading.
