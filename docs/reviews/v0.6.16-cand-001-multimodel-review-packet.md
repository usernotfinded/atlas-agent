- `atlas agent autonomous-paper` — a deterministic, paper-only, local-first autonomous decision loop.

- `docs/autonomous-paper-loop.md` — design and safety-boundary documentation.

No live trading, live submit, broker order submission, real provider execution, source/package version bump, tag, GitHub Release, or PyPI publication is included.

## Model-specific review prompts

### GPT-5.5 — Architecture / product risk

Review the architecture and product-risk posture of CAND-001. Focus only on files changed in commit `e27357b`.

1. Does the autonomous paper loop remain cleanly separated from live broker adapters, live submit paths, and real provider execution? Confirm there is no accidental bridging code between `src/atlas_agent/agent/autonomous_paper.py` and broker submit APIs.
2. Is the scope appropriate for a planning candidate? Verify the change is paper-only, local-first, deterministic, and does not alter protected runtime boundaries under `src/atlas_agent/config`, `brokers`, `execution`, `safety`, or `risk`.
3. Does the design avoid product claims that could be read as live-trading readiness, profit guarantees, or risk elimination? Check `docs/autonomous-paper-loop.md`, `docs/shadow-live-readiness-contract.md`, and `CHANGELOG.md` for cautious wording.
4. Are the artifact format and evidence flow reviewable and redaction-safe? Confirm decision records and manifests do not encourage storing credentials or private financial data.
5. Identify any architectural overreach that should be split out of CAND-001 before it could be considered for the next planning gate.

### Claude Opus 4.6 Thinking — Safety / security / governance

Review CAND-001 from a safety, security, and governance perspective. Treat every boundary as fail-closed until proven otherwise.

1. Confirm the hard invariants in `docs/bounded-live-autonomy-governance.md` remain intact. Specifically: live trading disabled by default, live submit disabled by default, provider output never execution authority, `RiskManager` mandatory and deterministic, kill switch mandatory, approval queues mandatory where applicable, and audit hash-chain present.
2. Verify `run_autonomous_paper_loop` cannot be coerced into live mode. Examine how `AtlasConfig`, CLI flags, and `RiskLimits` are constructed; check for any path where `live_trading_enabled` or `can_submit` could become `true`.
3. Check that no secrets, credentials, API keys, or private broker state are loaded, logged, or committed by the new code or tests. Confirm `redact_payload` is applied to audit events.
4. Review the shadow-live contract (`docs/shadow-live-readiness-contract.md`) for clarity that shadow live is read-only, planning-only, must not submit orders, and must not mutate broker state.
5. Assess whether the static checkers (`scripts/check_autonomous_paper_loop_contract.py`, `scripts/check_shadow_live_contract.py`) enforce the documented boundaries deterministically without network access, credentials, or subprocess execution.
6. Flag any governance gap that would allow future shadow-live implementation to bypass the staged autonomy ladder or existing approval gates.
