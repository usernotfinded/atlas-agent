# Atlas Agent v0.6.x Roadmap

This roadmap moves Atlas Agent from a release-hardened trading framework toward a research, learning, reporting, and strategy platform. Each batch is scoped to one subsystem and is intended to be independently releasable.

## v0.6.0 - Backtesting Framework Foundation

- Objective: Replace the inline buy-and-hold-only backtest branch with a strategy framework that supports metadata, discovery, validation, benchmark abstraction, metrics abstraction, and a local strategy test harness.
- Architecture changes: Add a `BacktestStrategy` protocol, strategy metadata model, registry/discovery layer, built-in buy-and-hold strategy, benchmark abstraction, metrics calculator wrapper, validation harness, and CLI commands for strategy inspection.
- Files expected to change: `src/atlas_agent/backtest/*`, `src/atlas_agent/cli.py`, `tests/backtest/*`, `tests/cli/test_backtest_cli.py`, `tests/test_backtest_metrics.py`, `docs/releases/v0.6.0.md`, `CHANGELOG.md`.
- Migration risk: Low to medium. Existing `atlas backtest run --strategy buy_and_hold` behavior is preserved, while unknown strategies now fail closed instead of silently producing no strategy orders.
- Testing requirements: Registry tests, strategy loading tests, validation tests, metrics tests, CLI tests, existing backtest engine tests, and required safety checks.
- Release criteria: `atlas backtest list-strategies`, `atlas backtest describe buy_and_hold`, and `atlas backtest validate buy_and_hold` work locally; default backtest remains deterministic and local-first; broker, provider, live trading, risk, approval, kill-switch, audit hash-chain, and manifest boundaries remain unchanged.
- Estimated complexity: Medium.

## v0.6.1 - Backtesting Strategy Pack

- Objective: Add the first built-in strategy pack and local benchmark configuration while keeping strategies pluggable.
- Architecture changes: Add `moving_average_cross`, `rsi_mean_reversion`, typed strategy parameter specs, strategy parameter coercion, backtest run configuration fields, CLI parameter overrides, and a local-only SPY benchmark abstraction.
- Files expected to change: `src/atlas_agent/backtest/*`, `src/atlas_agent/config/schema.py`, `src/atlas_agent/cli.py`, `configs/strategy.example.yaml`, `templates/routine-trader/configs/strategy.example.yaml`, `tests/backtest/*`, `tests/cli/test_backtest_cli.py`, docs.
- Migration risk: Medium. Existing `buy_and_hold` remains registered and keeps its deterministic order ID format; invalid parameters now fail closed before simulation.
- Testing requirements: Full strategy-pack tests, registry tests, strategy loading tests, parameter validation tests, benchmark tests, config default tests, CLI tests, existing engine tests.
- Release criteria: `buy_and_hold`, `moving_average_cross`, and `rsi_mean_reversion` list and validate locally; parameterized backtests run locally; SPY benchmark requires explicit local benchmark CSV data; broker/provider/live execution boundaries remain unchanged.
- Estimated complexity: Medium.

## v0.6.2 - Research Artifact to Backtest Bridge

- Objective: Let local research artifacts propose paper-only strategy hypotheses for backtesting without creating orders or approvals.
- Architecture changes: Add a read-only bridge from research artifacts to backtest strategy inputs and local evaluation reports.
- Files expected to change: `src/atlas_agent/research/*`, `src/atlas_agent/backtest/*`, `tests/research/*`, `tests/backtest/*`, docs.
- Migration risk: Medium. Must preserve provider execution boundaries and avoid direct AI-to-broker execution.
- Testing requirements: Artifact parsing tests, paper-only bridge tests, provider safety regression tests, CLI tests.
- Release criteria: Bridge reads local artifacts only, creates no pending orders, and cannot submit broker orders.
- Estimated complexity: Medium.

## v0.6.3 - Backtest Reporting Layer

- Objective: Add structured reports for strategy runs, benchmarks, diagnostics, and audit-friendly rejected orders.
- Architecture changes: Add report models, Markdown/JSON writers, report index metadata, and deterministic output paths.
- Files expected to change: `src/atlas_agent/backtest/report.py`, `src/atlas_agent/reports/*`, tests, docs.
- Migration risk: Low. Reporting is additive.
- Testing requirements: Snapshot-style report tests, JSON schema tests, rejected-order auditability tests, artifact hygiene checks.
- Release criteria: Backtest reports are local, deterministic, free of secrets, and include strategy and benchmark metadata.
- Estimated complexity: Medium.
- **Status:** Local report generator foundation implemented. Supports daily, weekly, and ad-hoc Markdown/JSON reports using real local data (portfolio, backtest, research, risk, audit, system health). See `docs/reports.md`.

## v0.6.4 - Learning Loop Observability

- Objective: Add local learning-loop summaries that connect research notes, backtest results, and strategy validation outcomes.
- Architecture changes: Add read-only aggregation models and memory-safe summary artifacts.
- Files expected to change: `src/atlas_agent/learning/*`, `src/atlas_agent/reports/*`, `tests/test_memory_index.py`, learning tests, docs.
- Migration risk: Medium. Must avoid storing secrets or presenting outcomes as advice.
- Testing requirements: Memory safety tests, summary tests, redaction tests, no-profit-claim checks.
- Release criteria: Summaries are local, auditable, redacted, and do not alter execution paths.
- Estimated complexity: Medium.
- **Status:** Reflection artifact foundation implemented. Supports structured local reflection artifacts with draft/pending_review/approved/rejected/archived states, static fallback (provider disabled by default), provenance/audit metadata, and CLI lifecycle commands. See `docs/reflection.md`.
- **Status:** Skill candidate foundation implemented. Supports structured skill candidate artifacts derived from reflections or local files, with draft/pending_review/approved/rejected/archived/promoted states, manual-only promotion into a local skill library, provenance/audit metadata, and CLI lifecycle commands. See `docs/skills.md`.
- **Status:** Learning suggestion foundation implemented. Supports structured learning suggestion artifacts derived from reflections, skills, or local files, with draft/pending_review/accepted/rejected/archived states, advisory-only execution policy, provenance/audit metadata, and CLI lifecycle commands. See `docs/learning-loop.md`.

## v0.6.5 - Strategy Comparison Workspace

- Objective: Compare multiple local strategy backtests against benchmarks with shared data and consistent metrics.
- Architecture changes: Add comparison result models, multi-run orchestration, and CLI reporting.
- Files expected to change: `src/atlas_agent/backtest/*`, `src/atlas_agent/reports/*`, tests, docs.
- Migration risk: Medium. Must keep each run deterministic and avoid shared mutable state.
- Testing requirements: Multi-strategy tests, metric consistency tests, CLI tests, deterministic artifact tests.
- Release criteria: Comparisons run locally, preserve individual run diagnostics, and do not change live/paper trading defaults.
- Estimated complexity: Medium to high.

## v0.6.6 - Dashboard Research Read-Only Views

- Objective: Display strategy, backtest, and research report summaries in the existing dashboard without adding secrets or write actions.
- Architecture changes: Add dashboard collectors and read-only render sections for local artifacts.
- Files expected to change: `src/atlas_agent/dashboard/*`, `tests/dashboard/*`, `docs/dashboard.md`, docs/security/dashboard-security.md`.
- Migration risk: Low to medium. Dashboard must remain strictly read-only.
- Testing requirements: Dashboard collector tests, render tests, zero-secret regression tests, no-action-control tests.
- Release criteria: Dashboard exposes only local read-only summaries and cannot submit, approve, or mutate trading state.
- Estimated complexity: Medium.
- **Status:** Dashboard data layer and static read-only UI enhancement implemented. The dashboard renders local sections for system health, safety status, portfolio, backtests, reports, reflections, skills, learning suggestions, audit/events, warnings, and missing data. It remains local-only, static, provider-disabled, broker-disabled, and free of trading, provider, broker, skill activation, and learning execution controls. See `docs/dashboard.md` and `docs/security/dashboard-security.md`.

## v0.6.7 - Broker Roadmap and Guarded Adapter Status

- Objective: Formalize broker support status and add fail-closed tests and guards for unsupported or partial broker adapters without changing live trading or live submit defaults.
- Architecture changes: Add a static broker support inventory (`BrokerSupportEntry`), fail-closed guard helpers (`guard_submit`, `guard_sync`), read-only `atlas broker status` CLI command, and comprehensive tests proving unsupported/disabled/placeholder brokers fail closed.
- Files expected to change: `src/atlas_agent/brokers/status.py`, `src/atlas_agent/brokers/guards.py`, `src/atlas_agent/brokers/__init__.py`, `src/atlas_agent/cli.py`, `tests/brokers/test_broker_status.py`, `tests/brokers/test_broker_guards.py`, `tests/brokers/test_unsupported_brokers_fail_closed.py`, `tests/cli/test_brokers_cli.py`, `tests/fixtures/cli_command_contract.json`, `tests/fixtures/product_capability_inventory.json`, `docs/broker-roadmap.md`, `docs/brokers.md`, `docs/v0.6-roadmap.md`, `CHANGELOG.md`.
- Migration risk: Low. Changes are additive; no live submit behavior is changed. Alpaca live submit remains gated by existing opt-in, credentials, kill switch, and config checks.
- Testing requirements: Broker inventory tests, guard tests, fail-closed tests for unsupported/disabled/placeholder brokers, CLI status tests, existing broker tests must continue passing.
- Release criteria: `atlas broker status` works locally without credentials; `guard_submit` and `guard_sync` reject unsupported/disabled/placeholder brokers; PaperBroker remains default; Alpaca behavior is unchanged; Binance remains partial; CCXT remains disabled; IBKR remains placeholder; no live trading or live submit defaults change.
- Estimated complexity: Low to medium.
- **Status:** Broker support inventory and guarded adapter status implemented. See `docs/broker-roadmap.md`.
