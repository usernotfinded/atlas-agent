# Release v0.5.7.dev27 — Real Provider Integration Threat Model & Policy Draft

## Summary

This release is **documentation and policy only**. No real provider execution is added. No network calls. No API keys. No provider SDKs. No broker changes.

Batch 8.0 introduces a formal threat model, provider execution policy draft, integration requirements checklist, and architecture decision record for future real provider integration. These artifacts define the safety boundaries, acceptance criteria, and policy rules that must be satisfied before any real LLM/API provider call can be implemented.

## What Changed

### New Documentation

- `docs/security/provider-integration-threat-model.md`
  - Scope, assets, trust boundaries, threat categories (prompt injection, API key leakage, excessive agency, artifact tampering, etc.).
  - 10 concrete attack scenarios with mitigations.
  - Required controls before real provider execution (opt-in policy, manual unlock, credential isolation, payload redaction, response validation, audit events, dry-run parity, rollback mechanism).
  - Acceptance criteria for future provider execution batch.

- `docs/security/provider-execution-policy.md`
  - Default deny: provider execution is disabled by default.
  - Human opt-in: no automatic state transitions to enable provider calls.
  - Credential policy: API keys never stored in artifacts, logs, or CLI output.
  - Outbound payload policy: bounded, redacted, minimized payloads only.
  - Provider response policy: responses are untrusted, denylist-clean, and require review.
  - Trading separation policy: provider output is not a trade signal.
  - Audit policy: every call requires call-plan, dry-run, state, audit, readiness, freeze, request hash, response hash, timestamps.
  - Failure policy: fail closed on missing credentials, unsafe payload, drift, or unsafe response.
  - Provider allowlist policy: no provider enabled by default; explicit adapter registration.
  - Release gate policy: new focused gates required before provider integration release.

- `docs/security/provider-integration-requirements.md`
  - Checkbox checklist organized by phase: before adapter interface, before credential loading, before first network call, before accepting response, before trading pipeline integration.
  - Required tests, required docs, required manual review points, required rollback/revoke behavior.

- `docs/adr/ADR-0001-provider-execution-boundary.md`
  - Decision: provider execution remains isolated from trading execution.
  - Consequences: more artifacts, slower integration, stronger safety, easier auditability.
