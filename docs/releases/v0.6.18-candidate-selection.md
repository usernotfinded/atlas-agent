## Proposed candidates

- **CAND-010:** Safety-State Persistence Regression Guard.

### Why CAND-010 is eligible

- It is a static regression guard only; it does not implement, authorize, or
  enable any trading or execution path.
- It hardens the CAND-009 safety-state persistence hardening by turning the
  manual fixed-temp write grep into a committed, named, tested, and integrated
  checker.
- It scans only the four safety persistence modules that CAND-009 migrated and
  allows `atomic_write.py` as the unique `mkstemp`-based temporary helper.
- It introduces no live trading, live submit, broker/provider execution,
  credential loading, order placement, pending-order creation, network access,
  or approval queue mutation.
- It preserves CAND-009 runtime behavior and all existing fail-closed
  guarantees.
- It includes deterministic checker tests and is wired into dev, CI, and quick
  release gates.
- The release cutover bumps the source/package version to `0.6.18`, creates an
  annotated `v0.6.18` tag, creates a GitHub Release, and does not publish to
  PyPI.

## Candidate acceptance

**CAND-010** is accepted into the `v0.6.18` candidate chain with acceptance
verdict `PASS` on 2026-07-01. `v0.6.18` was released on 2026-07-01 after a
separate release-readiness review. This acceptance does not change runtime
behavior beyond the static regression guard already landed on `main`, nor does
it enable live trading or live submit, authorize order placement, load
credentials, call brokers or providers, access the network, or mutate runtime
state or approval queues. It is not an approval to trade, not a live-readiness
claim, and not permission to submit orders.

### Future work

- No candidates beyond CAND-010 are proposed for `v0.6.18` at this time.

### Selection rules

CAND-010 satisfies the repository's candidate selection rules:

1. No live trading enablement — fail-closed defaults unchanged.
2. No broker/provider execution enablement — no direct AI-to-broker path.
3. No secrets or credentials committed.
4. No unsafe marketing claims — no promises of specific returns, eliminated
   risk, live readiness, or autonomous trading readiness.
5. Protected runtime boundaries preserved — changes are confined to a static
   source-text checker and its tests.
6. Small candidate batch — reviewable and independently valuable.
7. Checker/test/doc coverage — deterministic static guard with companion tests.
8. Green CI before completion — passes targeted verification commands.
9. Release cutover, if any, will be performed only after a separate
   release-readiness review and will preserve all safety boundaries: no live
   trading enablement, no live submit, no order placement, no broker/provider
   execution, no credential loading, no network access, and no PyPI publication.
