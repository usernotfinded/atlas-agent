## Summary

`v0.6.23` is a release-maintenance tooling release. It includes
**CAND-017: Post-Release CI Run-ID Recorder**, a docs/checker/test-only
candidate that adds `scripts/update_release_assurance_ci.py` and tests. The
script queries GitHub Actions for core workflow runs associated with a release
tag and injects the run IDs and URLs into the post-release assurance dossier
(`docs/releases/v0.6.X-post-release-assurance.md`) and its optional JSON mirror.
It is dry-run by default and requires `--write` to mutate files. It uses the
authenticated `gh` CLI and Python standard library only.

This release directly addresses the post-release assurance gap observed after
the `v0.6.22` cutover: the tag-triggered CI runs completed after the dossier was
committed, leaving the CI section as a placeholder that had to be updated
manually.

This is a **GitHub-only release**. PyPI was not published. No live trading, live
submit, broker order submission, provider execution, credential loading, order
placement, pending-order creation, or approval queue mutation is introduced or
enabled. `atlas run --mode live` remains fail-closed.

## Included Candidates

- **CAND-017: Post-Release CI Run-ID Recorder**
  - Adds `scripts/update_release_assurance_ci.py` and
    `tests/test_update_release_assurance_ci.py`.
  - Verifies the GitHub Release exists via `gh release view`.
  - Lists workflow runs for the tag via `gh run list` and filters to the core
    workflows (`CI`, `Release Gate`, `Atlas Agent Paper Routines`), keeping the
    most recent run per workflow.
  - Generates a markdown table under `## GitHub Actions / CI Status` and a JSON
    array under `ci_status.runs`.
  - Dry-run by default; only mutates files when passed `--write`.
  - Markdown edits are bounded to the `## GitHub Actions / CI Status` section;
    the JSON file is updated atomically (write to temp, rename).
  - Uses only the Python standard library and the authenticated `gh` CLI,
    performs no network mutation, loads no credentials, and does not change
    runtime, safety, broker, provider, or CLI behavior.
  - Does not change `RiskManager`, kill-switch, deadman, heartbeat, or audit
    hash-chain behavior, and does not enable live trading, live submit,
    broker/provider execution, credential loading, network access, or approval
    queue mutation.
  - Does not broaden the CAND-014 provider-artifact extraction boundary, which
    remains strictly at
    `src/atlas_agent/research/provider_mock_response_final_safety_seal.py`.

## Safety Boundaries

- This release does not enable live trading or live submit.
- This release does not enable provider execution.
- No autonomous trading is claimed or enabled by this release.
- This release does not place, cancel, or flatten orders.
- This release does not create pending orders or mutate approval queues.
- This release does not call brokers, providers, or network endpoints.
- This release does not load credentials, secrets, or API keys.
- `RiskManager`, approval gates, the kill switch, the dead-man heartbeat, and
  the audit hash-chain remain unchanged and are not weakened.
- `atlas run --mode live` remains fail-closed (exits 2).
- CAND-017 is a static helper/test/docs addition only; no runtime safety
  decision changes.
- No live-readiness, autonomous-live-readiness, trading safety, broker
  endorsement, or permission to submit orders is claimed.
- No profitability, absolute-safety, or future-performance claim is made.

## Known Non-Blocking Warnings

1. A full `pytest` run or full `bash scripts/release_check.sh` may exceed local
   interactive timeouts. The `--quick` gate and all targeted checks pass.
2. Historical candidate-chain docs for pre-`v0.6.23` releases still emit
   expected warnings from `check_candidate_chain.py` because they reflect
   historical planning state rather than current metadata.
