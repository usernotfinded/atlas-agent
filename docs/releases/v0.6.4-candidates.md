# v0.6.4 Patch Candidate Selection

> **Status:** planning — candidate selection only
> **Last updated:** 2026-06-06
> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

## Status

`v0.6.3` is the current public GitHub release (tagged and published). `v0.6.4` is **not**
version-prepared, **not** tagged, and **not** released. This document is a planning artifact
that selects safe maintenance candidates for a future `v0.6.4` patch release. It does **not**
authorize implementation, version bumps, tag creation, GitHub release creation, or PyPI
publishing. `docs/releases/v0.6.4.md` does not exist yet and must not be created in this batch.

## Selection Criteria

A candidate is eligible for `v0.6.4` only if **all** of the following are true:

1. **Small and safe** — changes are docs-only, checker-only, test-only, or CI-only.
2. **Backward compatible** — no API changes, no CLI contract changes, no breaking behavior.
3. **No runtime trading impact** — does not modify broker, execution, safety, risk, or provider defaults.
4. **Patch scope** per Semantic Versioning 2.0.0 — backward-compatible bug fixes or small maintenance only.
5. **Verified by evidence** — the issue is reproducible from current repo state, CI logs, or checker output.
6. **Testable** — the fix can be covered by existing or new tests without network or credentials.

Candidates that fail any criterion are rejected or deferred to a later minor release.

## Accepted Candidates

### CAND-001 — `main_health.py` drift prevention after release cutovers
- **What:** Improve the version-aware constants in `scripts/main_health.py` so the
  post-release update path is obvious and less error-prone; update matching tests.
- **Why:** After every release, `EXPECTED_SOURCE_VERSION`, `PUBLIC_RELEASE`, and
  `NEXT_UNREQUESTED_RELEASE_TAG` must be updated manually. Missing one causes
  `main_health.py` to fail on an otherwise clean post-push state.
- **Files:** `scripts/main_health.py`, `tests/test_main_health.py`
- **Safety:** Checker/test-only. No runtime impact.
- **Status:** Implemented in Batch 60.

### CAND-002 — Release-assurance artifact hygiene follow-up
- **What:** Clarify generated-artifact checker messages and docs guidance for
  `artifacts/release_assurance/*` folders produced by `scripts/release_assurance.py`.
- **Why:** Release-assurance packs are local-only evidence. When they are left
  untracked they can trigger `working_tree_dirty` warnings and confuse operators.
- **Files:** `scripts/check_generated_artifacts.py`, `docs/development/checks-reference.md`,
  `README.md`
- **Safety:** Checker/docs-only. No runtime impact.
- **Status:** Implemented in Batch 62.

### CAND-003 — Package distribution checker follow-up after `v0.6.3`
- **What:** Harden `--dry-run` path and missing-tool error handling in
  `scripts/check_package_distribution.py`.
- **Why:** The v0.6.3 release was partly motivated by package-distribution checker
  fixes. A small follow-up makes the checker more robust for local development.
- **Files:** `scripts/check_package_distribution.py`, `tests/test_package_distribution_check.py`
- **Safety:** Checker/test-only. No runtime impact.
- **Status:** Implemented in Batch 61.

### CAND-004 — Release notes / trust center consistency after `v0.6.3`
- **What:** Final consistency pass across release notes, trust center, and checker
  docs to remove any remaining `v0.6.2` references and align examples.
- **Why:** Batch 58 updated most docs, but a few checker examples and cross-references
  may still reference the previous release.
- **Files:** `docs/trust/v0.6.3-status.md`, `docs/development/checks-reference.md`,
  `docs/development/main-health.md`, `docs/release-checklist.md`
- **Safety:** Docs-only. No runtime impact.
- **Status:** Implemented in Batch 63.

### CAND-006 — CI deterministic test reliability follow-up
- **What:** Improve isolation and deterministic ordering for tests that compare
  artifact JSON or depend on filesystem iteration order.
- **Why:** Flaky tests undermine CI confidence. Deterministic ordering and better
  mocks remove a class of false negatives without resorting to retries or skips.
- **Files:** `tests/test_release_assurance.py`, `tests/test_main_health.py`, and
  relevant research tests.
- **Safety:** Test-only. No runtime impact.
- **Status:** Implemented in Batch 63.

### CAND-007 — Generated artifact cleanup ergonomics
- **What:** Enhance `scripts/check_generated_artifacts.py` to emit copy-paste-ready
  backup commands for untracked local artifacts.
- **Why:** Manual backup is error-prone. Clear instructions reduce operator friction
  while preserving the policy of never auto-deleting evidence.
- **Files:** `scripts/check_generated_artifacts.py`
- **Safety:** Checker-only. No destructive behavior.
- **Status:** Implemented in Batch 61.

### CAND-009 — PyPI non-publish messaging consistency
- **What:** Align wording across release notes and public docs that state PyPI
  publish was not performed.
- **Why:** Consistent negated messaging makes the no-publish policy unambiguous
  to reviewers and automated forbidden-claims scanners.
- **Files:** `docs/releases/*.md`, `README.md`, `SECURITY.md`
- **Safety:** Docs-only. No runtime impact.
- **Status:** Selected for implementation in a future batch.

### CAND-010 — `v0.6.4` release prep checker planning
- **What:** Add a skeleton `scripts/check_v064_release_prep.py` and matching tests
  that reject premature `v0.6.4` release artifacts while the version remains `0.6.3`.
- **Why:** Following the established pattern prepares the release-prep gate so it
  can be activated immediately after the version bump without last-minute scaffolding.
- **Files:** `scripts/check_v064_release_prep.py`, `tests/test_v064_release_prep.py`
- **Safety:** Checker/test-only. Rejects rather than enables release actions.
- **Status:** Selected for implementation in a future batch.

## Rejected / Out-of-Scope Candidates

The following items are explicitly rejected for `v0.6.4`:

- **Live trading default changes** — Atlas remains disabled-by-default for live trading.
- **Live submit enablement** — Live submit remains disabled by default.
- **Broker execution enablement** — No broker adapter may be promoted to default live-submit enabled.
- **Provider execution unlock** — Provider adapters remain fail-closed and disabled by default.
- **Autonomous trading** — Not claimed or enabled.
- **Automatic skill activation** — Skill candidates remain manually reviewed and offline.
- **Automatic learning execution** — Learning workflows remain offline and manually triggered.
- **PyPI publish** — Not approved in this planning batch; requires separate process.
- **Major dashboard rewrite** — Out of patch scope.
- **Large architecture rewrite** — Out of patch scope and too risky.
- **New broker adapters** — New features, not maintenance.
- **New strategy packs** — New features, not maintenance.
- **Profitability or trading correctness claims** — Forbidden.
- **Financial advice wording** — Forbidden.

## Safety Boundaries

Regardless of which candidates are accepted later, these boundaries remain
non-negotiable for any `v0.6.4` work:

- `src/atlas_agent/config/` — no new live defaults.
- `src/atlas_agent/brokers/` — no new submit paths, no execution enabled by default.
- `src/atlas_agent/execution/` — no direct AI-to-broker paths.
- `src/atlas_agent/safety/` — no kill switch bypasses.
- `src/atlas_agent/risk/` — risk limits must not be weakened.
- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
- ❌ Version is **not** bumped to `0.6.4` in this batch.
- ❌ `docs/releases/v0.6.4.md` release notes file does **not** exist yet.
- ❌ No tag, GitHub release, or PyPI publish yet.
