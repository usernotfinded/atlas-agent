## Released candidates

- **CAND-017: Post-Release CI Run-ID Recorder** — released on 2026-07-13 as
  part of `v0.6.23`. A docs/checker/test-only candidate that adds
  `scripts/update_release_assurance_ci.py` and tests to update post-release
  assurance dossiers with GitHub Actions run IDs. It is dry-run by default and
  requires `--write` to mutate files. It uses the authenticated `gh` CLI and
  Python standard library only. It changes no runtime, safety, CLI, broker,
  provider, credential, version, or release-metadata behavior and does not
  broaden the CAND-014 boundary. Design and plan:
  [docs/superpowers/specs/2026-07-13-cand-017-post-release-ci-recorder-design.md](../superpowers/specs/2026-07-13-cand-017-post-release-ci-recorder-design.md)
  /
  [docs/superpowers/plans/2026-07-13-cand-017-post-release-ci-recorder.md](../superpowers/plans/2026-07-13-cand-017-post-release-ci-recorder.md).

### Why CAND-017 was eligible

- It is a docs/checker/test-only addition; it does not implement, authorize, or
  enable any trading or execution path.
- It closes a concrete release-maintenance gap observed after the `v0.6.22`
  cutover: post-release assurance dossiers can be automatically populated with
  GitHub Actions run IDs once the tag-triggered workflows complete.
- It uses only the Python standard library and the already-authenticated `gh`
  CLI, performs no network mutation, loads no credentials, and does not mutate
  repository state unless explicitly passed `--write`.
- It preserves the CAND-014 single-module extraction boundary.
- It ships deterministic, offline tests.

## Selection rules

Any future `v0.6.24` candidate must satisfy the repository's candidate selection
rules before acceptance:

1. No live trading enablement — fail-closed defaults unchanged.
2. No broker/provider execution enablement — no direct AI-to-broker path.
3. No secrets or credentials committed.
4. No unsafe marketing claims — no promises of specific returns, eliminated
   risk, live readiness, or autonomous trading readiness.
5. Protected runtime boundaries preserved — `RiskManager`, approval queues, kill
   switch, heartbeat, deadman, audit hash-chain, live-submit opt-in, and
   `can_submit` are not bypassed.
6. Small, reviewable candidate batches that are independently valuable.
7. Deterministic test and documentation coverage.
8. Green verification before acceptance.
9. The CAND-014 provider-artifact extraction boundary is not broadened beyond
   `src/atlas_agent/research/provider_mock_response_final_safety_seal.py`.
