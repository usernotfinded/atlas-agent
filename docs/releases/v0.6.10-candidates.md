# v0.6.10 Candidate Planning Baseline

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

> **Status:** release-prep — all selected candidates implemented. This document does **not** authorize
> tag creation, GitHub release creation, or PyPI publishing. Those steps require explicit owner approval in separate batches.

`v0.6.9` is the current public GitHub release (tagged and published). `v0.6.10` is the
next planned release. This document selects the safe, conservative candidates for the
`v0.6.10` patch cycle and records the evidence gathered from the post-release scout pass.

## Context

The v0.6.9 release focused on deterministic local backtesting, the backtest report schema
contract, validation UX, and Markdown report readability. The README was streamlined and
provider badge/list corrections landed in `docs/model-providers.md`.

The v0.6.10 batch continues that hygiene-only theme: reviewer-facing docs/checker
hardening, dashboard/report UX polish, backtest schema checker ergonomics, template and
code-inventory hygiene, and release-assurance metadata cleanup. All candidates are
local-only, deterministic, and offline; no runtime trading, broker, provider, execution,
risk, approval, or kill-switch behavior changes.

## Selection criteria

A candidate is eligible for `v0.6.10` only if **all** of the following are true:

- It improves reviewer trust, report clarity, checker ergonomics, or docs accuracy.
- It is a small, backward-compatible docs/checker/test/report improvement.
- It does not change runtime trading, broker, provider, risk, approval, or kill-switch behavior.
- It does not add new runtime features, broker adapters, or provider integrations.
- It does not require new secrets, credentials, or network calls.
- It is safe to implement directly on `main` without a feature branch.
- It does not weaken existing checks or safety boundaries.

## Safety boundaries

- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Risk gates remain deterministic and unchanged.
- Approval queue behavior remains unchanged.
- Kill-switch behavior remains unchanged.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
