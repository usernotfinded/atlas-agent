## Proposed candidates

- **CAND-012:** Candidate-Chain Consistency Guard.

### Why CAND-012 was eligible

- It is a docs/checker/test-only addition; it does not implement, authorize, or
  enable any trading or execution path.
- It adds deterministic validation that release metadata, candidate-chain JSON,
  and candidate-chain Markdown remain aligned, coherent, and free of premature
  release or live-trading claims.
- It adds `scripts/check_candidate_chain.py`, `tests/test_candidate_chain.py`,
  dev/CI integration, and checks-reference documentation.
- It uses only the Python standard library, performs no network access, loads no
  credentials, imports no `atlas_agent` runtime code, and does not mutate
  repository files.
- It introduces no live trading, live submit, broker/provider execution,
  credential loading, order placement, pending-order creation, network access,
  or approval queue mutation.
- It changes no `RiskManager`, kill-switch, deadman, heartbeat, or audit
  hash-chain code.
- It includes deterministic tests and passes the targeted verification commands.

## Candidate acceptance

**CAND-012** is accepted into the `v0.6.20` candidate chain with acceptance
verdict `PASS_WITH_WARNINGS` on 2026-07-01. This acceptance does not change
runtime behavior beyond the checker/test/docs already landed on `main`, nor does
it enable live trading or live submit, authorize order placement, load
credentials, call brokers or providers, access the network, or mutate runtime
state or approval queues. It is not an approval to trade, not a live-readiness
claim, and not permission to submit orders.

### Selection rules

CAND-012 satisfies the repository's candidate selection rules:

1. No live trading enablement — fail-closed defaults unchanged.
2. No broker/provider execution enablement — no direct AI-to-broker path.
3. No secrets or credentials committed.
4. No unsafe marketing claims — no promises of specific returns, eliminated
   risk, live readiness, or autonomous trading readiness.
5. Protected runtime boundaries preserved — changes are confined to a static
   checker, its tests, dev/CI wiring, and documentation.
6. Small candidate batch — reviewable and independently valuable.
7. Test/doc coverage — deterministic behavior-preservation tests and updated
   candidate-chain documentation.
8. Green verification before acceptance — passes targeted verification commands.
