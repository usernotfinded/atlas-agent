## Proposed candidates

- **CAND-009:** Safety State Atomic-Write Hardening.

### Why CAND-009 is eligible

- It is a safety-state persistence hardening change only; it does not enable or
  modify any trading or execution path.
- It replaces fixed `<target>.tmp` writes with unique same-directory temporary
  files, removing temp-name collision risk while preserving atomic `replace`
  semantics and fail-closed behavior.
- It preserves file formats, JSON indentation/sorting, public APIs, and
  best-effort `chmod(0o600)` behavior.
- It introduces no live trading, live submit, broker/provider execution,
  credential loading, order placement, pending-order creation, or approval queue
  mutation.
- It includes deterministic helper tests, concurrency tests, and safety-module
  regression tests.
- It resolves a CLI contract hygiene warning by documenting the existing
  read-only `doctor` command.
- The release cutover bumps the source/package version to `0.6.17`, creates an
  annotated `v0.6.17` tag, creates a GitHub Release, and does not publish to
  PyPI.

## Candidate acceptance

**CAND-009** is accepted into the `v0.6.17` candidate chain with acceptance
verdict `PASS` on 2026-06-30. `v0.6.17` was released on 2026-07-01 after a
separate release-readiness review. This acceptance does not change runtime
behavior beyond the safety-state write implementation already landed on `main`,
nor does it enable live trading or live submit, authorize order placement, load
credentials, call brokers or providers, or mutate runtime state. It is not an
approval to trade, not a live-readiness claim, and not permission to submit
orders.

### Selection rules

CAND-009 satisfies the repository's candidate selection rules:

1. No live trading enablement — fail-closed defaults unchanged.
2. No broker/provider execution enablement — no direct AI-to-broker path.
3. No secrets or credentials committed.
4. No unsafe marketing claims — no promises of specific returns, eliminated
   risk, live readiness, or autonomous trading readiness.
5. Protected runtime boundaries preserved — changes are confined to safety-state
   persistence mechanics.
6. Small candidate batch — reviewable and independently valuable.
7. Checker/test/doc coverage — helper and regression tests are deterministic.
8. Green CI before completion — passes targeted verification commands.
9. Release cutover, if any, will be performed only after a separate
   release-readiness review and will preserve all safety boundaries: no live
   trading enablement, no live submit, no order placement, no broker/provider
   execution, no credential loading, and no PyPI publication.

This candidate selection remains planning-only and is not a release.
