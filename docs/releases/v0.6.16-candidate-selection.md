## Proposed candidates

- **CAND-001:** Paper Autonomous Decision Loop and Shadow-Live Readiness Contract.
- **CAND-002:** Autonomous Paper Decision Quality Scorecard and Promotion Gate.

### Why CAND-001 is eligible

- It keeps live trading and live submit disabled by default.
- It does not enable broker order submission or real provider execution.
- It adds only paper-only, local-first, deterministic runtime behavior.
- It introduces a planning-only shadow-live/read-only contract that explicitly
  forbids order submission and broker-state mutation.
- It includes deterministic static checkers and tests that enforce the
  contract.
- It does not bump the source/package version, create a tag, cut a GitHub
  Release, or publish to PyPI.

### Why CAND-002 is eligible

- It builds only on existing paper-only artifacts produced by CAND-001.
- It is deterministic, offline, and does not call brokers, providers, or load
  credentials.
- It adds a fail-closed promotion gate that defaults to `blocked`.
- It produces explicit machine-readable and human-readable evidence for future
  shadow-live/read-only review decisions.
- It includes deterministic static checkers and tests that enforce the
  contract.
- It does not bump the source/package version, create a tag, cut a GitHub
  Release, or publish to PyPI.

### Why CAND-003 is eligible

- It remains entirely inside paper mode; live trading, live submit, and broker
  submission are still disabled.
- It adds no provider calls or credential loading; the stateful runner is
  local-first and works offline.
- The execution-neutral kernel is a simulation boundary, not a path to live
  execution.
- Resume and duplicate-prevention are paper-only conveniences; they do not
  implement shadow-live or mirror a real account.
- Configurable costs and next-bar fills produce honest metrics without claiming
  live performance, execution quality, or readiness.
- It includes updated documentation and safety boundaries; no source/package
  version bump, tag, GitHub Release, or PyPI publication.

### Why CAND-004 is eligible

- It evaluates only paper artifacts and rejects live-mode or live-referencing inputs.
- It adds no broker, provider, or live execution calls and loads no credentials.
- It produces conservative, fail-closed quality states (`blocked` by default when evidence is insufficient).
- It extends the CAND-002 promotion gate with trading-quality dimensions without enabling shadow-live or live trading.
- It includes deterministic static checkers and tests that enforce the contract.
- It does not bump the source/package version, create a tag, cut a GitHub Release, or publish to PyPI.

### Why CAND-005 is eligible

- It is strictly read-only and fixture-first; it calls no real broker APIs, loads no credentials, and submits no orders.
- It builds only on existing paper artifacts and the CAND-004 trading-quality gate.
- It produces deterministic, fail-closed comparison statuses and both JSON and Markdown artifacts.
- It includes a static contract checker and tests that enforce read-only boundaries and forbid live-submit patterns.
- It does not claim live readiness, profitability, or eliminated risk.
- It does not bump the source/package version, create a tag, cut a GitHub Release, or publish to PyPI.

### Why CAND-006 is eligible

- It is a simulated-only conformance rehearsal; it does not submit orders, call
  brokers or providers, load credentials, instantiate runtime trading objects,
  mutate state, or indicate live readiness.
- It consumes only existing CAND-004 and CAND-005 artifacts plus hypothetical
  order-intent and simulated kill-switch, risk-envelope, and approval fixtures.
- It produces deterministic, non-transmittable dry-run submit evidence in both
  JSON and Markdown forms.
- It includes a static contract checker and tests that enforce the
  simulated-only boundary and forbid live-submit patterns.
- It does not bump the source/package version, create a tag, cut a GitHub
  Release, or publish to PyPI.

### Future work

- No candidates beyond CAND-006 are proposed for `v0.6.16`.

### Selection rules

CAND-001, CAND-002, CAND-003, CAND-004, CAND-005, and CAND-006 satisfy the repository's candidate selection rules:

1. No live trading enablement — fail-closed defaults unchanged.
2. No broker/provider execution enablement — no direct AI-to-broker path.
3. No secrets or credentials committed.
4. No unsafe marketing claims — no promises of specific returns, eliminated
   risk, live readiness, or autonomous trading readiness.
5. Protected runtime boundaries preserved — changes under `src/atlas_agent/config`,
   `brokers`, `execution`, `safety`, and `risk` are avoided.
6. Small candidate batch — reviewable and independently valuable.
7. Checker/test/doc coverage — every candidate has deterministic coverage.
8. Green CI before completion — passes `dev_check.sh`, `ci_check.sh`, and
   `release_check.sh --quick`.
9. No release cutover side effects — no tag, GitHub Release, PyPI, or source
   version bump.

This proposal remains planning-only until the release is explicitly cut.
