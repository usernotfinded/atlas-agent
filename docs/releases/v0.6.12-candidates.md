# v0.6.12 Candidate Planning Baseline

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

> **Status:** release-prep — CAND-001 through CAND-016 are implemented/audited; CAND-008 and CAND-010 are not separately tracked (folded) and disposition is closed. This document does **not** authorize tag creation, GitHub release creation, or PyPI publishing; those require a separate owner-approved cutover batch.

`v0.6.11` is the current public GitHub release (tagged and published). `v0.6.12` is the next planned release. This document selects the conservative, hygiene-only candidates that were completed for the `v0.6.12` planning cycle.

## Context

The `v0.6.12` release line continues the hygiene-only trajectory established by `v0.6.10` and `v0.6.11`: reviewer-facing documentation, deterministic checkers, release-assurance hardening, trust-center transparency, and archival/artifact hygiene. All `v0.6.12` candidates are local-only, deterministic, and offline; none change runtime trading, broker execution, provider execution, risk gates, approval behavior, or kill-switch behavior.

The batch focuses on consolidating product-demo readiness, reviewer trust snapshots, release-assurance bundles and diagnostics, legacy document archival, and artifact retention auditing. These are docs, checker, workflow, and test additions only.

## Selection criteria

A candidate is eligible for `v0.6.12` only if **all** of the following are true:

- It improves reviewer trust, release transparency, checker coverage, or docs accuracy.
- It is a small, backward-compatible docs/checker/test/workflow improvement.
- It does not change runtime trading, broker, provider, risk, approval, or kill-switch behavior.
- It does not add new runtime features, broker adapters, or provider integrations.
- It does not require new secrets, credentials, or network calls.
- It is safe to implement directly on `main` without a feature branch.
- It does not weaken existing checks or safety boundaries.

## Safety boundaries

- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Risk gates remain deterministic and unchanged.
- Approval queue behavior remains unchanged.
- Kill-switch behavior remains unchanged.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
