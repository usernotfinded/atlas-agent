# v0.6.6 Patch Candidate Selection

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

> **Status:** planning — candidate selection only

`v0.6.5` is the current public GitHub release (tagged and published). `v0.6.6` is **not**
version-prepared, **not** tagged, and **not** released. This document is a planning artifact
that selects safe maintenance candidates for a future `v0.6.6` patch release. It does **not**
authorize implementation, version bumps, tag creation, GitHub release creation, or PyPI
publishing. `docs/releases/v0.6.6.md` does not exist yet and must not be created in this batch.

## Context

Batch 73 (post-v0.6.5 hygiene) already implemented the two candidates originally proposed
in `docs/releases/v0.6.6-plan.md`:

- **CAND-001** (plan) — Update public docs to reflect `v0.6.5` as current public release — **implemented in Batch 73**.
- **CAND-002** (plan) — Update `main_health.py` and tests to reflect `v0.6.5` as current public release — **implemented in Batch 73**.

This selection document therefore focuses on **new** candidates for the `v0.6.6` line.

## Selection criteria

A candidate is eligible for `v0.6.6` only if **all** of the following are true:

- It is a small, backward-compatible fix or docs improvement.
- It does not change runtime trading, broker, provider, risk, approval, or kill-switch behavior.
- It does not add new runtime features, broker adapters, or provider integrations.
- It does not require new secrets, credentials, or network calls.
- It is safe to implement directly on `main` without a feature branch.
- It does not weaken existing checks or safety boundaries.

## Safety boundaries

- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Risk gates remain deterministic and unchanged.
- Approval queue behavior remains unchanged.
- Kill-switch behavior remains unchanged.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
