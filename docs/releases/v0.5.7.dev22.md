# Release v0.5.7.dev22 — Provider Execution Replay Envelope Consistency

## Summary

Hardening release that fixes the semantic consistency of `atlas research provider-execution-replay` when the source provider call plan has changed since the dry-run was created.

## Behavior Change

### Before

When the source provider call plan was modified after dry-run creation, replay returned a generic error envelope:

```json
{
  "ok": false,
  "status": "provider_execution_dry_run_source_hash_mismatch"
}
```

### After

Replay now returns a consistent replay envelope with `match=false`, because a hash mismatch is a valid replay result, not a malformed command:

**Non-strict:**
```json
{
  "ok": true,
  "status": "research_provider_execution_dry_run_replayed",
  "match": false,
  "expected_hash": "...",
  "actual_hash": "...",
  "checks": [...],
  "warnings": [...]
}
```
Exit code: `0`

**Strict (`--strict`):**
Same JSON envelope, but exit code: `2`

## What Changed

- `safe_validate_provider_execution_dry_run_data()` accepts `for_replay: bool = False`. When True, skips the source call-plan hash match so replay can detect drift.
- `replay_provider_execution_dry_run()` loads with `for_replay=True`, then rebuilds and compares. Returns `warnings` including a drift warning when `match=false`.
- CLI handler includes `warnings` in JSON output.

## Safety

- No provider calls.
- No API keys read.
- No network requests.
- No live trading authorization.
- Tampered artifacts (impossible booleans, forbidden fragments, unsafe lineage) still fail safely with generic error envelopes.
- All configless invariants preserved.
