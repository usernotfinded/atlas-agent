# v0.6.8 Patch Candidate Selection

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

> **Status:** planning — CAND-001 implemented; CAND-002/CAND-003/CAND-004 accepted, not yet implemented

`v0.6.7` is the current public GitHub release (tagged and published). `v0.6.8` is the
next planning line. This document selects the safe public-demo-proof maintenance
candidates for the `v0.6.8` patch release. It does **not** authorize version bumps,
tag creation, GitHub release creation, or PyPI publishing. Those steps require
explicit owner approval in a separate release-prep batch.

## Context

Batch 83 (post-v0.6.7 test-performance and local-heat-reduction hygiene) completed
the tiered local check system (`scripts/smoke_check.sh`, `scripts/local_quick_check.sh`,
`docs/development/check-tiers.md`) and tightened CI determinism. `v0.6.7` is now
tagged, released, and current. The next planning line is `v0.6.8`.

This selection document focuses on **public demo proof and reviewer trust** — making
the paper demo easier to audit, verify, and trust without changing runtime trading,
broker, provider, execution, risk, approval, or kill-switch behavior.

A multi-agent swarm review (Phase 2) discovered three specific over-promise issues
in public-facing docs that directly undermine reviewer trust. These are folded into
the reviewer-path consolidation candidate because they affect the same surfaces
reviewers read.

## Selection criteria

A candidate is eligible for `v0.6.8` only if **all** of the following are true:

- It improves demo auditability, reviewer clarity, or public trust.
- It is a small, backward-compatible fix or docs/checker improvement.
- It does not change runtime trading, broker, provider, risk, approval, or kill-switch behavior.
- It does not add new runtime features, broker adapters, or provider integrations.
- It does not require new secrets, credentials, or network calls.
- It is safe to implement directly on `main` without a feature branch.
- It does not weaken existing checks or safety boundaries.

## Safety boundaries

- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Risk gates remain deterministic and unchanged.
- Approval queue behavior remains unchanged.
- Kill-switch behavior remains unchanged.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
