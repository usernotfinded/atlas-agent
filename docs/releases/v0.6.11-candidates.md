# v0.6.11 Candidate Planning Baseline

> **Not financial advice.** Atlas Agent is a software tool, not a financial advisor. Trading involves significant risk of loss.

> **Status:** planning — CAND-001 through CAND-004 implemented; remaining candidates are not implemented. This document does **not** authorize
> tag creation, GitHub release creation, PyPI publishing, or version bumps. Those steps
> require a separate owner-approved release-prep and cutover batch.

`v0.6.10` is the current public GitHub release (tagged and published). `v0.6.9` is
historical. This document selects safe, conservative candidates for the `v0.6.11`
planning cycle. CAND-001 and CAND-002 are implemented; the release remains in planning.

## Context

The v0.6.10 release continued the hygiene-only theme from v0.6.9: reviewer-facing
docs/checker hardening, dashboard/report UX polish, backtest schema checker ergonomics,
template source-of-truth simplification, code-inventory follow-ups, release-assurance
metadata cleanup, and public docs link/reference hardening.

The v0.6.11 batch continues that conservative hygiene trajectory: post-release
hardening, observability cleanup, documentation polish, checker simplification, test-suite
performance, dashboard/export refinements, and trust-center/public-docs updates. All
candidates are local-only, deterministic, and offline; no runtime trading, broker
execution, provider execution, risk, approval, or kill-switch behavior changes.

## Selection criteria

A candidate is eligible for `v0.6.11` only if **all** of the following are true:

- It improves reviewer trust, report clarity, checker ergonomics, or docs accuracy.
- It is a small, backward-compatible docs/checker/test/report improvement.
- It does not change runtime trading, broker, provider, risk, approval, or kill-switch behavior.
- It does not add new runtime features, broker adapters, or provider integrations.
- It does not require new secrets, credentials, or network calls.
- It is safe to implement directly on `main` without a feature branch.
- It does not weaken existing checks or safety boundaries.

## Safety boundaries

- Live trading remains disabled by default.
- Live submit remains disabled by default.
- Provider execution remains disabled by default.
- Broker execution remains disabled by default.
- Risk gates remain deterministic and unchanged.
- Approval queue behavior remains unchanged.
- Kill-switch behavior remains unchanged.
- Human approval remains required for live order flow.
- Credentials are not loaded unless explicitly configured.
- No provider APIs, broker APIs, or real webhooks are called from CI or local checks.
